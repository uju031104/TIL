### Class 와 Struct의 차이점에 대해서 이야기 해주세요

C++에서 Class와 Struct는 기본 접근 제어자의 설정을 제외하면 거의 동일합니다.
Struct의 기본 접근 제어자는 public이고 Class는 private입니다.

Struct는 데이터를 묶을 때 사용합니다. 데이터를 다루는 아주 최소한의 함수만 사용합니다.
반면, Class는 객체의 행동(Behavior)과 캡슐화가 중요할 때 사용합니다. 또한 상속 구조가 복잡하거나 인터페이스가 필요할 때 사용합니다.

언리얼 엔진에서는 이 차이가 조금 더 엄격해집니다.

USTRUCT는 액터의 변수로 포함시키는 용도로 사용하며 가비지 컬렉션 대상이 아닙니다. 따라서 가볍습니다.
UCLASS는 액터, 컴포넌트 등 핵심 객체들을 만들 때 사용합니다. 가비지 컬렉션에 의해 관리되며 메모리 오버헤드가 더 큽니다.

### new와 malloc의 차이점에 대해서 이야기해 주세요

new는 연산자입니다. operator new 라는 특수 함수로 동작합니다. (메모리 할당 -> 생성자 호출의 과정) 
반환 포인터, 크기 등을 알아서 계산해줍니다.
메모리가 부족할 때 예외처리(bad_alloc)를 합니다.

malloc은 단순한 함수입니다. 생성자 호출은 없고 메모리만 할당합니다.
malloc의 문법을 보면 알수있듯 형 변환도 직접 해줘야하고 크기도 직접 계산해줘야 합니다.
할당 실패시 NULL을 반환해서 무조건 NULL 체크를 해줘야합니다.



operator new 라는 특수 함수로 동작하는데 이 함수는 오버로딩으로 커스텀 할 수 있다. (전역 범위/클래스 범위 둘 다 가능)

new 오버로딩 하는이유

이미 할당된 메모리(buffer) 위에 객체만 생성하고 싶을 때

-> 메모리 풀인데 메모리를 미리 크게 할당하고 조금씩 나눠서 사용하는 기법

사용하는 이유: 
메모리 파편화(Memory Fragmentation)방지 - 조금씩 하면 메모리 사이 공간이 생겨버려서 비효율적임
속도 향상 - 메모리 할당을 할 때 시스템 콜(System Call)을 동반해서 비용이 크다. 한 번만 해놓으면 비용이 적어서 빠름.

구현 -> 큰 메모리를 하나 할당하고 작은 조각으로 나눈다.
연결 리스트로 조각들을 연결한다.
할당하면 메모리 리스트에서 뺀다.
해제하면 다시 메모리 리스트에 넣어준다.

스택은 높은 주소 -> 낮은 주소  / 힙은 낮은 주소 -> 높은 주소 
메모리를 효율적으로 쓰기 위해 나눠짐

특정 행동(함수)을 하기 위해 OS에게 요청하는 것이 시스템 콜(System Call)이다.

### OOP에 대해서 설명해 보세요. (중요!)

객체 지향 프로그래밍은 데이터와 그 데이터를 처리하는 함수를 묶어서 하나의 객체로 두고 관리하는 방식입니다.(패러다임)   
객체지향 프로그래밍(OOP)의 4대 핵심 요소는 다음과 같습니다.     
캡슐화, 상속, 다형성, 추상화입니다    

캡슐화 (Encapsulation): 데이터와 기능을 하나로 묶고 외부로 부터 보호합니다. 객체가 외부에서 변경되는것을 막기 위함입니다.   
상속 (Inheritance): 기존 클래스의 기능을 물려받아 재사용하고 확장하는 것. 계층 구조가 된다. (사람은 동물이다.)   
다형성 (Polymorphism): 하나의 인터페이스나 함수가 상황에 따라 다르게 동작하는 것. 오버라이딩과 오버로딩이 있다.      
추상화 (Abstraction): 복잡한 내부 구현은 숨기고, 사용자에게 필요한 최소한의 인터페이스(기능)만 노출하는 것     

오버로딩은 정적 바인딩, 오버라이딩은 동적 바인딩이다.   

**바인딩(Binding)**이란 식별자(함수 이름 등)와 그 실제 메모리 주소를 연결하는 과정을 의미   

오버로딩은 컴파일 타임에 미리 바인딩을 해둬서 빠르고 안정적이다.   
컴파일러가 함수 이름, 매개 변수 타입과 개수를 보고 미리 정하는 것이다.   
-> virtual이 없는 일반 멤버 함수는 다 이렇다.   

오버라이딩은 런타임(프로그램 실행중) 호출될 함수를 결정한다.   
v-table(가상 함수 테이블)을 참조하다보니 정적바인딩에 비해서 상대적으로 느리다.   
부모 클래스 포인터로 여러 자식 객체의 기능을 사용할 수 있다.   

**friend 키워드**   

기본적으로 클래스의 private이나 protected 멤버는 외부에서 접근할 수 없습니다. 하지만 클래스 내부에서 특정 대상(함수나 다른 클래스)을 friend로 지정하면, 그 대상은 해당 클래스의 모든 멤버에 자유롭게 접근할 수 있게 됩니다.   

```cpp
// 연산자 오버로딩(가장 흔한 사례)
class Player {
private:
    int Health = 100;

public:
    // 외부 함수인 operator<< 가 private 멤버인 Health에 접근할 수 있게 허용
    friend std::ostream& operator<<(std::ostream& os, const Player& p);
};

std::ostream& operator<<(std::ostream& os, const Player& p) {
    os << "Player Health: " << p.Health; // friend 덕분에 접근 가능!
    return os;
}
```


### 순수 가상 함수는 vtable에 어떻게 등록 되어있을까?
에러 처리 함수로 등록되어있습니다. 보통 __purecall (MSVC 기준) 또는  __cxa_pure_virtual(GCC/Clang 기준)이라는 함수의 주소가 등록됩니다.
이렇게 등록된 이유는 실수로라도 순수 가상 함수를 호출할 경우 명확한 에러 표시를 해주기 위함입니다.   

순수 가상 함수를 상속받은 자식이 재정의를 했다면, 해당 주소가 재정의한 실제 함수이다.   

한 걸음 더: 순수 가상 함수도 본체를 가질 수 있다?   
재미있게도 C++에서는 순수 가상 함수도 본체({ ... })를 정의할 수 있습니다.
단, 그렇더라도 vtable에는 여전히 __purecall이 들어갑니다. 이 본체는 자식 클래스들이 Base::Func(); 처럼 명시적으로 공통 기능을 호출할 때만 쓰이며, 다형성을 통한 직접 호출은 여전히 막힙니다.   

소멸자를 순수 가상함수로 만드는 경우?   
클래스에 순수 가상 함수가 하나도 없지만 해당 클래스를 추상화 하고싶은 경우 소멸자를 순수 가상함수로 만들어줍니다.
단, 소멸자가 순수 가상함수이면 외부 구현부에서 {} 빈칸이라도 꼭 구현은 해줘야합니다.
왜냐하면 소멸자는 자식 -> 부모 순으로 호출되는데 부모 소멸자가 구현이 안되어있으면 주소가 없어서 링킹 에러가 뜨기 때문입니다.
그래서 빈칸이라도 구현을 해줘서 주소 등록을 해줍니다.

### Vtable에 대해서 설명 해주세요.   

Vtable(Virtual Method Table)이란 가상 함수들의 주소값을 모아둔 정적인 배열입니다.   
클래스에 가상함수가 있다면 컴파일 타임에 Vtable이 생성되고 클래스 내부에 vptr라는 숨겨진 포인터 변수가 추가됩니다.   

객체가 생성될 때 vptr는 해당 클래스의 vtable를 가리킵니다.   

프로그램이 가상 함수를 호출하면 객체의 vptr를 확인하고 해당 vtable로 이동 후 함수를 실행합니다.    


**업캐스팅에서 vptr**   
업캐스팅에서 부모 포인터로 자식 객체를 가리킬때 가상함수를 실행하면 자식에서 재정의된 함수가 실행되는데 그 이유가 객체 마다 vptr이 저장되기 때문에 자식 객체는 자식 vtable을 가리킨다.   

**다중 상속시 vptr**   
부모 클래스 개수만큼 vptr이 존재한다.   
Thunk(썽크) :  

장점   
컴파일 타임이 아니라 실행 중에 객체의 실제 타입에 맞는 함수를 호출할 수 있어서 OOP의 핵심인 다형성을 완성합니다.   

단점   
각 객체마다 vptr 포인터 크기(64비트 기준 8바이트)만큼 메모리를 더 사용합니다. (메모리 오버헤드)   
함수를 직접 호출하지 않고 vptr -> vtable -> 함수 주소 라는 2단계를 거쳐서 미세하게 느리다. (성능 오버헤드)   
컴파일러 단계에서 미리 정하지 않기 때문에 inline 최적화가 어렵습니다.           
단, 포인터가 아닌 직접 객체를 호출하거나 final 키워드 붙이면 가능! 또는 포인터로 호출하더라도 컴파일러의 정적 분석을 통해 객체의 타입을 추론할 수 있는 경우(Devirtualization) 인라인화가 수행될 수 있습니다.   


일반 함수와 인라인 함수   
일반 함수 호출: 호출 시 CPU 레지스터 저장, 스택 프레임 생성, 주소 점프, 반환 시 복구 등 **오버헤드(Overhead)**가 발생합니다.   

인라인 함수: 컴파일러가 함수 호출 코드 자체를 함수의 실제 내용으로 치환합니다. 즉, 함수 호출이라는 단계 자체가 사라집니다. 짧고 자주 호출되는 함수에 사용하면 좋습니다.     

```cpp
inline int square(int x) {
    return x * x;
}

int main() {
    int a = 5;
    int result = square(a); // 컴파일 시 result = a * a; 로 치환될 수 있음
}
```

**인라인 함수의 장단점**  

장점   
성능 향상: 함수 호출 오버헤드가 사라져 실행 속도가 빨라집니다.   

최적화 기회: 코드가 호출 지점에 직접 삽입되므로, 컴파일러가 주변 코드와 연계하여 추가적인 최적화를 수행하기 좋습니다.   

단점   
코드 비대화 (Code Bloat): 함수가 여러 곳에서 호출되면 그만큼 전체 실행 파일의 크기가 커집니다.   

캐시 효율 저하: 코드 크기가 너무 커지면 CPU의 명령어 캐시(Instruction Cache) 적중률이 떨어져 오히려 성능이 하락할 수 있습니다.   

**주의사항**   
inline 키워드는 컴파일러에게 보내는 제안이라서 컴파일러가 이 요청을 무시할 수 있다.   

무시되는 경우: 함수가 너무 길거나, 재귀 함수이거나, 루프가 복잡하거나, 가상 함수(Virtual Function)인 경우 등입니다.   

자동 인라인: 클래스 내부(정의부)에 직접 작성한 멤버 함수는 inline 키워드를 쓰지 않아도 컴파일러가 알아서 인라인 후보로 등록합니다.     

**인라인 함수와 #define 매크로의 차이**

둘 다 코드를 치환한다는 점에선 비슷하지만 굉장히 다릅니다.   

**#define매크로**   
컴파일 전인 전처리기(Preprocessor)가 수행   
단순하게 텍스트 치환   
타입 체크 불가능   
디버깅 어려움(심볼 테이블에 없음)   
안정성이 낮음   

**inline함수**   
컴파일 단계에서 수행   
실제 코드 삽입 및 최적화   
타입 체크 가능(매개변수 타입)   
디버깅 쉬움(디버거가 인식 가능)   
안정함(일반 함수와 동일한 문법으로 실행)   

```cpp
#define SQUARE(x) ((x) * (x))
inline int square(int x) { return x * x; }

int a = 5;
SQUARE(++a); // ((++a) * (++a)) 가 되어 a가 두 번 증가함 (결과: 42 또는 49, 비정상)
square(++a); // a를 한 번 증가시킨 후(6) 함수 호출 (결과: 36, 정상)
```

### 소멸자를 Virtual로 만들어야 하는 이유는?

소멸자가 virtual이면 vtable을 참조하여 동적 타입인 자식 클래스의 소멸자를 먼저 호출한 후 부모 소멸자를 호출합니다.   
이는 업캐스팅된 포인터를 통해 객체를 삭제할때 발생하는 메모리 누수를 없애기 위함입니다.
(C++에서는 부모 포인터로 해제를 할 때 소멸자에 virtual이 없는 경우 미정의 동작이라고 규정하고 있습니다.)   
또한, 메모리 누수 말고도 자식 클래스가 가지고 있던 리소스들이 제대로 종료되지 않아 예기치 못한 오류가 발생할 수 있습니다.   

단, 상속할 일이 없으면 소멸자를 virtual로 만들 필요가 없습니다.    
이럴땐 클래스에 final 키워드를 붙여줍니다.   

### C++ 4대 캐스팅 방법에 대해 설명해 보세요.
**static_cast**   
가장 많이 사용하고 컴파일 타임에 결정된다.   
기본적인 데이터 타입 변환에 많이 사용된다.   
부모, 자식 클래스간의 변환도 되는데 안전성 검사는 없다.   
*void를 구체적인 타입으로 변환
런타임 오버헤드가 없지만, 안전성은 100% 보장하지 않는다.   
```cpp
float FloatValue = 3.14f;
// 실수에서 정수로 변환 (데이터 손실을 인지하고 있음을 명시)
int IntValue = static_cast<int>(FloatValue);

// 상속 관계에서의 업캐스팅 (자식 -> 부모)
// MyPawn은 APawn의 자식이므로 항상 안전함
APawn* BasePawn = static_cast<APawn*>(MyPawnInstance);
```

<br/>

**dynamic_cast**    
런타임에 상속관계에 있는 클래스간의 안전한 변환을 보장한다.   
주로 부모 클래스 포인터를 자식 클래스 타입으로 변환(다운캐스팅)할때 사용한다.   
RTTI(Runtime Type Information)을 사용하여 변환 가능 여부를 확인한다.   
변환이 불가능할 경우 포인터는 nullptr를 반환하고 참조형은 예외를 발생시킨다.   
반드시 하나 이상의 가상함수가 포함된 다형성 클래스여야 사용 가능하다.   
```cpp
// 가상 함수가 하나라도 있는 클래스여야 함
class Animal { virtual void Speak() {} };
class Dog : public Animal { void Speak() override { /* 멍멍 */ } };

Animal* MyAnimal = new Dog();

// MyAnimal이 실제 Dog인지 확인하며 캐스팅
Dog* MyDog = dynamic_cast<Dog*>(MyAnimal);

if (MyDog) {
    // 변환 성공 시에만 안전하게 접근
}
```

<br/>

**const_cast**   
객체의 const나 volatile 속성을 제거하거나 추가할 때 사용한다.   
주로 상수 포인터나 참조자를 비상수 포인터/참조자로 잠시 바꿔야 하는 경우에 사용한다.   
원래 const로 선언된 상수의 값을 이 연산자로 강제 변경하는 것은 정의되지 않은 동작(Undefined Behavior)을 유발할 수 있어서 주의해야한다.   

```cpp
void ModifyValue(const int* ReadOnlyValue) {
    // ReadOnlyValue의 const를 강제로 제거
    int* WritableValue = const_cast<int*>(ReadOnlyValue);
    *WritableValue = 100;
}
```

<br/>

**reinterpret_cast**      
서로 전혀 관계없는 타입 간의 비트 단위 재해석을 수행하는 강력하고 위험한 캐스팅이다.   
포인터를 정수형으로 변환하거나 반대의 상황에 사용   
서로 연관없는 클래스 포인터 간의 변환 
컴파일러는 어떠한 논리적 검사도 하지 않는다.   
하드웨어 제어나 특정 네트워크 프로토콜 처리 등 특수한 상황에서만 사용한다.  
```cpp
int Value = 65;
// 정수 65를 메모리 주소 0x41(65의 16진수)로 간주하고 포인터로 변환
int* Address = reinterpret_cast<int*>(Value);

// 특정 객체를 바이트 단위(char*)로 쪼개서 전송하거나 읽을 때
MyStruct Data;
char* RawData = reinterpret_cast<char*>(&Data);
```
